# Repository Guidelines

## Project Structure & Module Organization
- `src/main.js` bootstraps Vue 3, Element Plus, icon registry, router, global CSS, and exposes `$echarts`.
- `src/components` holds reusable UI (e.g., `HeaderCom.vue`, `FooterCom.vue`); `src/views` contains route-level pages.
- `src/router` defines navigation; update routes and lazy loading here.
- `src/api` centralizes Axios requests; prefer adding new endpoints here to keep usage consistent.
- `src/assets` stores styles and static media; `public` serves static root files; `dist/` is generated by builds and should remain untracked.

## Build, Test, and Development Commands
- `npm install` installs dependencies (Vue CLI 5, Element Plus, ECharts, Axios).
- `npm run serve` starts the dev server with hot reload; default port is Vue CLI’s standard unless overridden.
- `npm run build` creates a production bundle in `dist/`; use before packaging or deploying.
- `npm run lint` runs ESLint (Vue 3 essential + recommended rules); add `-- --fix` to auto-format common issues.

## Coding Style & Naming Conventions
- JavaScript/Vue: follow ESLint defaults in `package.json`; use 2-space indentation, semicolons, and single quotes for strings.
- Components and view files use PascalCase filenames (`HeaderCom.vue`); refs in templates stay in kebab-case.
- Keep imports grouped: vendor libs first, then local modules, then styles.
- Encapsulate view-specific styles with `<style scoped>`; place shared styles under `src/assets/css/`.

## Testing Guidelines
- No automated test suite is configured yet; rely on `npm run lint` plus manual UI checks per route.
- When adding tests, place them under a new `tests/` directory, name specs `*.spec.js`, and cover routing, API integration, and component rendering states.
- Document any new test commands in `package.json` scripts.

## Commit & Pull Request Guidelines
- Current history is minimal; write imperative, concise commit subjects (e.g., `Add search filter route`), and include scope when helpful.
- Reference related issues or tasks in the body; separate multiple changes into logical commits.
- Pull requests should summarize the change, list affected routes/components, include before/after screenshots for UI tweaks, and note how to reproduce and verify.
- Confirm `npm run lint` (and any added tests) before requesting review.

## Security & Configuration Tips
- Do not commit secrets or API keys; prefer Vue CLI env files (`.env.local`, `.env.development`) which stay untracked by default.
- Centralize base URLs and headers in `src/api` helpers so changes propagate consistently.
- Review third-party additions (Element Plus plugins, ECharts extensions) for license compatibility before inclusion.

## Agent Instructions (Frontend / Vue)

- **语言与输出**
  - 与作者交流时使用简体中文。
  - 在日志中展示文件内容时，尽量只展示关键片段和路径说明，
    不要长篇复制整个 `.vue` 文件，避免中文注释在终端显示乱码。

- **技术约束**
  - 前端基于 **Vue 3 + Vue Router + Element Plus + ECharts**。
  - 优先复用现有结构（`src/views`, `src/components`, `src/api`），避免大幅改动路由和布局。
  - HTTP 请求统一走 `src/api` 里的封装，不在组件中直接使用裸 `axios`/`fetch`。

- **代码风格**
  - 单文件组件保持现有命名方式（如 `HomePage.vue`, `HeaderCom.vue`）。
  - 组件内逻辑尽量拆分为函数或 composables，减少在 `<template>` 中写复杂表达式。
  - 适度增加注释，但避免在一个地方密集堆大量中文注释；
    建议把复杂逻辑的解释写在函数上方的简短注释中。

- **ChatBot 专属**
  - 修改 `ChatBot.vue` 时：
    - 保持现有 UI 结构（布局和主要交互不大改）。
    - 在修改对话逻辑/接口调用前，先用中文说明你的理解和设计思路。
    - 如需新增配置项/环境变量，请先说明新增字段的用途和默认值。

- **构建与测试**
  - 修改前端逻辑后，建议运行：
    - `npm run lint`
    - `npm run build`（如改动影响打包）
  - 在给出修改方案时，指出作者应该运行哪些命令来验证改动。
